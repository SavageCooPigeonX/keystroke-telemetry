## THE ACTUAL PROBLEM (NOT "split big files")

The auditor already finds oversized files. That's trivially solved. Any grep for `>50 lines` does that.

**The real problem is CHOOSING:**
- Which functions belong together vs. which separate?
- What shared state (constants, config, DB clients) resists splitting?
- Which imports create invisible coupling that breaks when you cut?
- Why some 400-line files RESIST extraction — what structural weakness holds them together?
- How do you know the extraction WORKED without re-auditing?

The `production_auditor/` already proved manual extraction works: one 831-line monolith became 34 files averaging 39 lines. But that took HUMAN JUDGMENT about every boundary. The Pigeon Compiler replaces that judgment with a **hardcoded state extractor** that maps the invisible bonds between functions and plans the cuts around them.

**Folder-Level PROMPT BOX** (auto-generated by `folder_auditor.py`):

This is injected into each folder's `MANIFEST.md` by `update_local_manifest()`. The function:
1. Calls `extract_tasks_from_audit()` which regex-parses `- [ ] **{ID}-NNN**: description` from DeepSeek output
2. Replaces or appends the `## 📦 PROMPT BOX` section in the local manifest
3. Also regenerates the `## 📁 FOLDER STRUCTURE` section with live line counts + 🔴/⚠️/🚫 markers

**The PROMPT BOX is the manifest writing itself.** DeepSeek audits → generates tasks → tasks get embedded into the manifest → next AI reads manifest → sees tasks → can execute them. This is the self-improving loop.

The `persist_pb_state()` function in `distributed_memory_seq002_v001.py` writes to DB, then `_sync_pb_to_master()` rebuilds the MASTER shard's PROMPT BOX table (3-column: `| PB | Status | Current Objective |`).

### 2. Pigeon Code — Deeper Than Rules

The last plan listed the rules. Here's what the rules actually MEAN in the codebase:

**The Sacred 8** (as enforced in `folder_auditor.py` prompt):
```
1. MAX 50 lines (preferred: 30) — PIGEON_MAX_LINES = 50, PIGEON_WARNING_LINES = 30
2. NO leading digits — breaks Python imports (can't `from 01_thing import x`)
3. Semantic naming: [category]_[description]_seqXXX_vXXX.py
4. One responsibility per file
5. Files must be rewritable WITHOUT referencing others
6. MANIFEST.md is source of truth (external memory)
7. Track line counts in manifest
8. Flag undocumented changes
```

**NOTE:** `master_auditor.py`'s prompt uses only 6 rules (not 8). Rules 5 and 8 are missing from its `PROJECT-WIDE RULES` block. Master adds its own rule 5: "Manifests should include Pigeon Code rules." The compiler should enforce all 8 in both prompts.

**Rule 3 decoded — seqXXX encodes EXECUTION ORDER:**
```
production_auditor/
├── model_config_seq001_v001.py      # Step 1: Load model configuration
├── prompt_builder_seq002_v001.py    # Step 2: Build prompts
├── json_parser_seq003_v001.py       # Step 3: Parse AI responses
├── snapshot_enrichment_seq004_v001.py # Step 4: Enrich snapshots
├── ...
├── pipeline_seq015_v001.py          # Step 15: Orchestrate everything
```

The sequence numbers ARE the pipeline. You can read the folder listing and understand the execution flow WITHOUT reading any code. This is what makes Pigeon Code self-documenting. The compiler must PRESERVE execution order in sequence assignments.

**Rule 5 decoded — "rewritable without referencing others":**
This means each file carries its own imports and has no implicit dependencies. If you delete every other file in the folder, any single file should still be UNDERSTANDABLE (even if it can't run). This is the constraint that makes extraction HARD — you have to duplicate imports and sometimes extract shared state into a `_config_` or `_constants_` file.

**The Gold Standard — production_auditor/ (34 files, avg 39 lines):**
This IS the proof. It was extracted from `runner/audit_orchestrator.py` (831 lines) by hand. Every design decision the compiler needs to make was already made here:
- `model_config_seq001_v001.py` (56 lines) — config constants extracted to their own file
- `pipeline_seq015_v001.py` (50 lines) — orchestrator imports from seq001-seq014, calls them in order
- `consensus_runner_seq008_v001.py` (52 lines) — one responsibility (run consensus)
- `__init__.py` — re-exports `run_full_audit()` so external callers don't change

### 4. The Ether Map — Invisible Dependency Graph

"Ether map" = the invisible medium through which imports and function calls flow between files and folders. The codebase already builds partial ether maps:

**`build_cross_reference()` in folder_auditor.py:**
```python
def build_cross_reference(folder: Path) -> str:
    imports = extract_imports(folder)       # regex: ^from X import / import X
    functions = extract_functions(folder)    # regex: ^def/class/async def name
    
    cross_ref = "## CROSS-REFERENCE MAP\n"
    cross_ref += "### Imports by File:\n"
    for file, imps in imports.items():
        cross_ref += f"  **{file}**: {', '.join(imps)}\n"
    cross_ref += "### Definitions by File:\n"
    for file, defs in functions.items():
        cross_ref += f"  **{file}**: {', '.join(defs)}\n"
    cross_ref += "### Internal Cross-References:\n"
    # tracks which files import from which other files in the same folder
```

**What's missing from the ether map:**
1. **Cross-FOLDER references** — `extract_imports()` only scans within one folder. But `production_auditor/pipeline_seq015_v001.py` imports from `integrations/`, `config/`, `storage_maif/`, `consensus/`. The compiler needs PROJECT-WIDE import tracing.
2. **Call graph depth** — Current extraction only gets `def function_name`. It doesn't trace which functions CALL which other functions (the call graph). For split decisions, this is critical: functions that call each other must stay together or get explicit cross-imports.
3. **Shared state detection** — Which functions share module-level variables, which classes share instance state, which files share DB connections from `config.database`. Shared state = resistance to splitting.
4. **Dead import detection** — Imports that exist but are never used in any function body. These are noise that confuse the dependency graph.
5. **Re-export chains** — `__init__.py` files re-export from sub-modules. The ether map must trace through these.

**What the compiler's ether map must produce:**
```
ETHER_MAP: codebase_auditor/folder_auditor.py
═══════════════════════════════════════════════

INBOUND (who imports from this file):
  codebase_auditor/__init__.py → audit_folder, audit_all
  codebase_auditor/master_auditor.py → (none, uses subprocess)

OUTBOUND (what this file imports):
  integrations.ai_deepseek_seq005_v001 → deepseek_query
  pathlib → Path
  re → compile, MULTILINE, finditer
  datetime → datetime
  argparse → ArgumentParser
  json → (unused? check)

INTERNAL CALL GRAPH:
  main() → audit_folder() → run_audit() → [list_folder_contents, read_all_files,
    load_local_manifest, load_master_manifest, load_previous_audit,
    build_cross_reference, deepseek_query]
  main() → audit_all() → audit_folder() (loop)
  audit_folder() → update_audit_results()
  audit_folder() → update_local_manifest() → [extract_tasks_from_audit,
    generate_folder_structure → [get_file_line_count, validate_pigeon_filename]]

SHARED STATE:
  PROJECT_ROOT = Path(__file__).parent.parent  # module-level constant
  PIGEON_MAX_LINES = 50                        # module-level constant
  PIGEON_WARNING_LINES = 30                    # module-level constant
  SKIP_FOLDERS, SKIP_FILES, AUDITABLE_FOLDERS  # module-level sets/lists

RESISTANCE POINTS (why this file stays big):
  1. run_audit() builds a massive f-string prompt (~96 lines of string template)
     → This string IS the function. Can't split a string literal across files.
     → Solution: move prompt template to separate file, import as constant.
  2. update_local_manifest() does string surgery on existing markdown
     → Tightly coupled to manifest format. If format changes, this breaks.
     → Solution: extract to manifest_writer file, keep format knowledge in one place.
  3. 6 helper functions (validate, count, list, read, extract×2) are only used
     by run_audit() but have no coupling to each other
     → Solution: group by purpose (pigeon_validators, file_readers, cross_ref_builder)
```

## THE HARDCODED STATE PROBLEM

This is the core insight the last plan missed entirely.

**Why files stay oversized despite everyone knowing the rules:**

### Resistance Pattern 1: The Prompt Template
`folder_auditor.py` is 633 lines. ~96 of those are a single f-string prompt template inside `run_audit()` (L262-358). You can't split a string literal across files without making it worse (joining strings from imports is uglier than one long string).

**Solution:** Extract prompt to `audit_prompt_template_seq002_v001.py` as a constant:
```python
# audit_prompt_template_seq002_v001.py (~50 lines — the prompt + format placeholders)
FOLDER_AUDIT_PROMPT = """You are a senior code auditor...
🐦 PIGEON CODE ENFORCEMENT — SACRED RULES:
..."""
```
The function in the runner file becomes `prompt = FOLDER_AUDIT_PROMPT.format(folder_name=..., contents=...)`.

### Resistance Pattern 2: Shared Module-Level State
`master_auditor.py` has `create_client()` called in `get_db_stats()` — the only function that accesses the DB directly. But the pattern matters: if you split into 12 files and MORE functions need DB access later, do they all inline `create_client()`?

**Solution:** Yes. Pigeon Rule 5: "Files must be rewritable without referencing others." Each file carries its own imports. Duplication of import lines is EXPECTED and DESIRED. The alternative (shared state file that all import from) creates a hidden coupling point.

### Resistance Pattern 3: The Orchestrator That Calls Everything
`pipeline_seq015_v001.py` in production_auditor imports from 14 other files and calls them in sequence. It's 50 lines of pure import + function calls. This is the CORRECT pattern — the orchestrator is thin, all logic lives in the files it imports.

**But what if the orchestrator itself needs >50 lines?** Then the pipeline has too many steps and needs to be broken into sub-pipelines (seq015a, seq015b) or the steps need consolidation.

### Resistance Pattern 4: The String Surgery Function
`update_local_manifest()` does string manipulation on markdown content — finding sections by header text, slicing and replacing. It's ~50 lines of `if "## 📦 PROMPT BOX" in existing:` / `find()` / slice logic. This function RESISTS splitting because every line depends on the parsing state from previous lines.

**Solution:** Don't split the function. Accept that some functions ARE 50 lines. The Pigeon limit is per-FILE not per-function. Put this function ALONE in its own file with its helper (`extract_tasks_from_audit`). Two functions, one responsibility (manifest writing), one file.

### Resistance Pattern 5: The Import Web
`maif_propaganda/routes.py` (925 lines) imports from 15+ modules. Every route handler needs different imports. Splitting into route files means each route file needs its own import block. But many routes share Flask decorators, session handling, DB client. This creates massive import duplication.

**Solution:** Accept the duplication. Alternatively, create a `routes_common_seq001_v001.py` that collects shared imports and re-exports them:
```python
# routes_common_seq001_v001.py
from flask import request, jsonify, session, redirect, url_for, flash
from config.database import get_db
from auth.decorators import login_required, admin_required
```
Then route files import from common: `from .routes_common_seq001_v001 import *`

## THE COMPILER ARCHITECTURE (Revised)

The compiler is a **3-layer system**:

### Layer 1: STATE EXTRACTOR (No AI — Pure AST + Regex)

This runs LOCALLY with zero API calls. It maps the hardcoded state of any Python file:

```
state_extractor/
├── __init__.py                              # (~10 lines)
├── ast_parser_seq001_v001.py                # (~45 lines) Python AST → function/class map
├── call_graph_seq002_v001.py                # (~40 lines) Which functions call which
├── import_tracer_seq003_v001.py             # (~45 lines) Full import chain resolution
├── shared_state_detector_seq004_v001.py     # (~40 lines) Module-level vars, class attrs
├── resistance_analyzer_seq005_v001.py       # (~45 lines) WHY this file resists splitting
├── ether_map_builder_seq006_v001.py         # (~50 lines) Full dependency graph → JSON
└── MANIFEST.md
```

**Output: The Ether Map JSON**
```json

  "file": "codebase_auditor/folder_auditor.py",
  "total_lines": 633,
  "functions": [...],
  "call_graph": {"main": ["audit_folder", "audit_all"], "audit_folder": ["run_audit", "update_audit_results", "update_local_manifest"]},
  "imports": {"inbound": [...], "outbound": [...]},
  "shared_state": ["PROJECT_ROOT", "PIGEON_MAX_LINES", "PIGEON_WARNING_LINES", "SKIP_FOLDERS"],
  "resistance_report": {
    "prompt_templates": [{"function": "run_audit", "string_lines": 80, "type": "f-string"}],
    "string_surgery": [{"function": "update_local_manifest", "type": "markdown_splice"}],
```

The resistance score (0.0-1.0) tells you HOW HARD this file will be to split. 0.0 = trivially splittable (just independent functions). 1.0 = monolithic (everything depends on everything). Most files score 0.3-0.6.

## THE HARDEST PART: CHOOSING BOUNDARIES

The user's exact words: "My biggest problem was choosing parameters n files that stay long"

This is what `resistance_analyzer_seq005_v001.py` solves. It classifies each oversized file into RESISTANCE PATTERNS:

### Pattern Classification

| Pattern | Score | Description | Solution |
|---------|-------|-------------|----------|
| **PROMPT_BLOB** | 0.3 | Large string templates inside functions | Extract string to constant file |
| **PURE_FUNCTIONS** | 0.1 | Independent functions, no shared state | One function per file (trivial) |
| **COUPLED_CLUSTER** | 0.5 | Functions that call each other | Group by call-graph clusters |
| **SHARED_STATE** | 0.6 | Functions sharing module-level variables | Extract state to config file |
| **STRING_SURGERY** | 0.7 | Functions doing text manipulation on same data | Keep together in one file |
| **FLASK_ROUTES** | 0.4 | Route handlers sharing decorators/blueprints | One route (or related group) per file, shared common file |
| **TEST_MONSOON** | 0.5 | Test files with 100+ test functions | Group by feature being tested |
| **IMPORT_WEB** | 0.8 | File imported by 10+ other files | Extract carefully, update all importers |
| **GOD_ORCHESTRATOR** | 0.9 | One function that calls everything | Reduce steps, or accept as thin orchestrator |

**Decision algorithm:**
```
1. Run AST parser → get functions, classes, constants
2. Run call graph → get who-calls-who
3. Run import tracer → get inbound/outbound dependencies
4. Run shared state detector → find module-level coupling
5. Classify each resistance pattern found
6. Compute aggregate resistance score
7. IF score < 0.3 → auto-extract (trivial, no AI needed)
8. IF score 0.3-0.7 → DeepSeek plans the cut (AI judgment needed)
9. IF score > 0.7 → flag for human review (too entangled, risky)
10. Generate cut plan with REASONS for each boundary decision
```

The REASONS matter. The operator (or future AI) needs to understand WHY functions were grouped. Not just "these go together" but "these share the `db` variable and both call `get_db()`, separating them would require passing `db` as a parameter to every function, which violates the 'rewritable without referencing others' rule."
